# /tsq-grill — 소크라틱 심층 인터뷰

질문을 통해 답을 끌어내는 소크라틱 방법론으로 PRD와 Sub-PRD를 작성한다.

PRD가 부실하면 그 위에 세워지는 모든 것이 부실해진다 — Sub-PRD, 설계, 구현, 테스트 전부.
그래서 최초 PRD부터 심도 있게 파고들어야 하고, 이 스킬이 그 역할을 한다.

## Mode Detection (자동)

```
.timsquad/ssot/prd.md 존재?
├── NO  → PRD Mode (프로젝트 전체 인터뷰)
└── YES → prd.md에 빈 섹션(TBD, TODO, 빈 테이블)이 있는가?
    ├── YES → PRD Reinforce Mode (빈 섹션 보강 인터뷰)
    └── NO  → Sub-PRD Mode (기능별 인터뷰)
```

사용자가 명시적으로 모드를 지정할 수도 있다:
- `/tsq-grill prd` → PRD Mode 강제
- `/tsq-grill sub` → Sub-PRD Mode 강제
- `/tsq-grill` → 자동 감지

---

## PRD Mode — 프로젝트 전체 인터뷰

PRD가 없거나 프로젝트 초기 단계에서 사용. 프로젝트의 본질을 파고든다.

### 인터뷰 원칙

아래 질문 트리는 **가이드**이지 체크리스트가 아니다.
사용자의 답변에 따라 유연하게 운용한다:

- 이미 답변된 영역은 건너뛴다
- 답변이 풍부한 영역은 더 깊이 파고든다
- 프로젝트 특성에 맞는 질문을 추가한다 (B2B라면 계약 모델, 모바일이라면 오프라인 지원 등)
- 한 번에 2-3개 질문만. 사용자가 부담을 느끼면 안 된다

### PRD-Phase 1: 비전 인터뷰

프로젝트 전체의 Why를 먼저 확립한다. 여기가 흔들리면 모든 후속 결정이 흔들린다.

```
Vision (비전)
├── 이 프로젝트를 한 문장으로 설명한다면?
├── 이게 없으면 세상에 어떤 문제가 남는가?
└── 6개월 후 성공했다면, 어떤 상태인가?

Target User (타겟)
├── 주요 사용자는 누구인가? (구체적 페르소나)
├── 그 사용자가 지금 이 문제를 어떻게 해결하고 있나?
├── 왜 기존 방법이 부족한가?
└── 사용자가 돈을 낼 의향이 있는 부분은?

Market (시장) — 경쟁 환경을 모르면 차별화도 불가능하다
├── 직접 경쟁자는? 간접 경쟁자는?
├── 경쟁자 대비 차별점은 무엇인가?
└── 타이밍이 지금인 이유는?
```

### PRD-Phase 2: 스코프 인터뷰

무엇을 만들고, 무엇을 만들지 않는지 경계를 확립한다.
스코프가 모호하면 나중에 반드시 크리프가 발생한다.

```
Core Features (핵심 기능)
├── 반드시 있어야 하는 기능 3-5개는?
├── 각 기능이 왜 필수인가? (없으면 어떻게 되나?)
├── Nice-to-Have로 미룰 수 있는 것은?
└── 명시적으로 만들지 않는 것은? (Anti-scope)

Technical Constraints (기술 제약)
├── 기술 스택이 이미 정해져 있나?
├── 외부 서비스/API 의존성은?
├── 성능 요구사항은? (응답시간, 동시접속 등)
└── 보안/규제 요구사항은? (인증, 결제, 개인정보)

Business Constraints (비즈니스 제약)
├── 예산/일정 제약은?
├── MVP 범위는 어디까지?
└── 런칭 후 첫 번째 마일스톤은?
```

### PRD-Phase 3: 성공 지표 인터뷰

정량적 측정 기준을 확립한다.
"성공"을 수치로 정의하지 못하면 완성 여부도 판단할 수 없다.

```
Metrics (지표)
├── 핵심 KPI는 무엇인가? (최대 3개)
├── 각 KPI의 현재 값과 목표 값은?
├── 측정 방법은? (어떤 도구/데이터로?)
└── 실패로 판단하는 기준은?
```

### PRD-Phase 4: PRD 생성

인터뷰 결과를 `.timsquad/ssot/prd.md`로 구조화:

1. 비전, 타겟, 시장 분석 → 배경 및 목적 섹션
2. 핵심 기능 목록 → **기능 인덱스 테이블** (Sub-PRD 링크는 빈 상태)
3. 기술/비즈니스 제약 → 제약사항 섹션
4. 성공 지표 → KPI 섹션
5. Anti-scope → 명시적 제외 범위 섹션
6. TBD 항목 → 미결 사항 테이블 (결정 시점 명시)

### PRD-Phase 5: 리뷰 & 전환 안내

1. 생성된 PRD를 사용자에게 리뷰 요청 (Human Checkpoint)
2. 승인 후 안내: "기능 인덱스의 각 기능을 `/tsq-grill`로 상세화하세요"

## PRD Reinforce Mode — 기존 PRD 보강

PRD가 존재하지만 빈 섹션(TBD, TODO, 빈 테이블)이 있을 때.

1. prd.md를 읽고 빈 섹션 / TBD 항목 목록을 사용자에게 제시
2. 사용자가 보강할 항목 선택
3. 해당 항목에 대해 PRD Mode와 동일한 소크라틱 인터뷰 진행
4. 결과를 prd.md에 반영
5. 사용자 리뷰 요청 (Human Checkpoint)

---

## Sub-PRD Mode — 기능별 인터뷰

PRD가 존재할 때 개별 기능을 깊이 파고들어 Sub-PRD를 작성한다.

### Sub-Phase 1: 대상 선정

1. `.timsquad/ssot/prd.md`의 기능 인덱스 테이블 읽기
2. Sub-PRD가 없거나 빈 기능 식별
3. 사용자에게 인터뷰 대상 기능 확인 (복수 선택 가능)

### Sub-Phase 2: 디자인 트리 인터뷰

기능마다 3단계 깊이로 질문한다. 한 번에 2-3개 질문, 답변 받으면 다음 깊이로.
PRD에서 확립한 비전/제약/KPI를 기반으로 질문한다 — PRD와 동떨어진 질문은 하지 않는다.

```
Why (목적)
├── 이 기능이 없으면 사용자에게 어떤 문제가 생기나?
├── 핵심 사용자 시나리오는?
└── 성공을 어떻게 측정하나?

What (범위)
├── Must-Have vs Nice-to-Have 경계는?
├── 이 기능이 다루지 않는 것은?
├── 어떤 데이터가 필요하고 어디서 오나?
└── 기존 시스템과 연동 포인트는?

How (설계)
├── 사용자 흐름 (Happy Path → Exception Path)?
├── 상태 전환이 있다면?
├── 에러 케이스와 처리 방식은?
└── 성능/보안 고려사항은?
```

### Sub-Phase 3: Sub-PRD 생성

인터뷰 결과를 `.timsquad/ssot/prd/[feature].md`로 구조화:

1. `ssot/prd/_template.md` 복사
2. 인터뷰 답변으로 각 섹션 채우기
3. Mapped Artifacts는 빈 링크로 두되 구조 유지
4. 설계 결정 테이블에 인터뷰 중 나온 선택지/결정 기록

### Sub-Phase 4: PRD 인덱스 업데이트

`prd.md`의 기능 인덱스 테이블에 새 Sub-PRD 링크 추가.

## 질문 방식 원칙 (공통)

이 원칙들은 인터뷰 품질을 좌우한다. 이유를 이해하고 따른다:

- **한 번에 질문 2-3개만** — 사용자가 질문 폭탄에 짧고 대충 답하게 되면 PRD 품질이 떨어진다
- **사용자 답변을 요약해서 확인받기** ("~라는 뜻이 맞나요?") — 에이전트의 해석이 맞는지 검증하는 단계. 이걸 건너뛰면 PRD에 사용자 의도가 아닌 에이전트의 추측이 들어간다
- **모호한 답변에는 구체적 예시로 재질문** — "사용자 경험 개선"은 PRD에 쓸 수 없다. "로그인 3초 내 완료"처럼 검증 가능한 수준까지 파고든다
- **"모르겠다"는 답변도 유효** — TBD로 기록하되 결정 시점을 명시한다. 억지로 답을 쥐어짜면 잘못된 전제가 된다
- **가정하지 않는다** — 선택지를 제시하고 사용자가 고르게 한다. 에이전트가 "아마 이럴 것이다"라고 채운 PRD는 위험하다

## Output

### PRD Mode
- `.timsquad/ssot/prd.md` — PRD 문서 (기능 인덱스 포함)
- 인터뷰 로그: `.timsquad/logs/grill-prd-[date].md`

### PRD Reinforce Mode
- `.timsquad/ssot/prd.md` — 보강된 PRD (기존 내용 유지, 빈 섹션만 채움)
- 인터뷰 로그: `.timsquad/logs/grill-prd-reinforce-[date].md`

### Sub-PRD Mode
- `.timsquad/ssot/prd/[feature].md` — Sub-PRD 파일
- `prd.md` 기능 인덱스 업데이트
- 인터뷰 로그: `.timsquad/logs/grill-[feature]-[date].md`

## Rules

이 규칙들이 존재하는 이유: 소크라틱 인터뷰의 핵심은 **사용자의 진짜 의도를 끌어내는 것**이지,
에이전트가 그럴듯한 문서를 만드는 것이 아니다.

- 사용자 답변을 임의로 해석하지 않는다 — 확인 후 기록. 잘못된 해석이 PRD에 들어가면 이후 모든 구현이 잘못된 전제 위에 세워진다
- "아직 모르겠다"는 TBD로 표시하되, 결정이 필요한 시점을 명시. 빈 칸보다 "Phase 2 시작 전까지 결정"이 낫다
- SSOT 외 파일 수정 금지
- PRD/Sub-PRD 생성 후 반드시 사용자에게 리뷰 요청 (Human Checkpoint)
