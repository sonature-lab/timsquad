### SP-001: SSOT 먼저, 코드 나중

| 항목 | 내용 |
|-----|------|
| **ID** | SP-001 |
| **카테고리** | DC (문서화) |
| **최초 발견** | - |
| **적용 횟수** | 0회 |
| **관련 에이전트** | tsq-planner, tsq-developer |
| **검증 상태** | standard |

**패턴 설명**
구현 전 반드시 SSOT 문서(service-spec.md, data-design.md)를 완성하고 승인받은 후 코드 작성을 시작한다.

**성공 요인**
- 명확한 계약서(Contract) 역할
- 프론트엔드-백엔드 병렬 작업 가능
- 리뷰 기준 명확

**적용 방법**
1. PRD 승인 확인
2. service-spec.md 작성 (API 명세)
3. data-design.md 작성 (DB 스키마)
4. error-codes.md 작성
5. 사용자 승인
6. Implementation phase 진입

**측정 가능한 효과**
- API 불일치 피드백 80% 감소
- 재작업 시간 60% 감소
- Level 2 피드백 50% 감소

**적용 확대**
- [x] 모든 프로젝트 타입에 기본 적용
- [ ] CI/CD에서 SSOT 검증 자동화

### SP-002: 구조화된 Handoff

| 항목 | 내용 |
|-----|------|
| **ID** | SP-002 |
| **카테고리** | CL (협업) |
| **최초 발견** | - |
| **적용 횟수** | 0회 |
| **관련 에이전트** | 전체 |
| **검증 상태** | standard |

**패턴 설명**
에이전트 전환 시 workspace.xml의 handoff 섹션을 완전하게 작성하여 컨텍스트 손실을 방지한다.

**성공 요인**
- 다음 에이전트가 즉시 작업 시작 가능
- 컨텍스트 재구성 토큰 절약
- 결정 사항 추적 가능

**적용 방법**
1. 작업 완료 시 workspace.xml 업데이트
2. handoff 섹션에 다음 정보 기록:
   - from/to 에이전트
   - 전달 메시지 (특이사항, 주의점)
   - 참조할 SSOT 문서
   - action-items (우선순위 포함)
3. 다음 에이전트 호출 전 검증

**측정 가능한 효과**
- 컨텍스트 재구성 시간 90% 감소
- 토큰 사용량 30% 절감
- 작업 연속성 향상

**적용 확대**
- [x] handoff 템플릿 표준화
- [ ] 자동 handoff 검증 스크립트

### SP-003: 점진적 검증 (Progressive Validation)

| 항목 | 내용 |
|-----|------|
| **ID** | SP-003 |
| **카테고리** | CQ (코드 품질) |
| **최초 발견** | - |
| **적용 횟수** | 0회 |
| **관련 에이전트** | tsq-developer, tsq-qa |
| **검증 상태** | validated |

**패턴 설명**
큰 기능을 작은 단위로 나누어 각 단계마다 테스트와 검증을 수행한다.

**성공 요인**
- 조기 문제 발견
- 디버깅 범위 축소
- 롤백 용이

**적용 방법**
1. 기능을 독립적인 단위로 분해
2. 각 단위별로:
   - 테스트 먼저 작성 (TDD)
   - 구현
   - 테스트 통과 확인
   - 커밋
3. 통합 테스트 실행
4. QA 리뷰

**측정 가능한 효과**
- 버그 발견 시간 70% 단축
- 수정 범위 80% 축소
- 테스트 커버리지 90%+ 유지

**적용 확대**
- [ ] CI에서 단위 테스트 필수화
- [ ] 커밋당 테스트 커버리지 검증

### SP-004: ADR 기반 의사결정

| 항목 | 내용 |
|-----|------|
| **ID** | SP-004 |
| **카테고리** | DC (문서화) |
| **최초 발견** | - |
| **적용 횟수** | 0회 |
| **관련 에이전트** | tsq-planner |
| **검증 상태** | validated |

**패턴 설명**
주요 아키텍처 결정은 ADR(Architecture Decision Record)로 문서화하여 결정 근거를 추적 가능하게 한다.

**성공 요인**
- 결정 배경 명확화
- 나중에 "왜 이렇게 했지?" 방지
- 대안 검토 기록 보존

**적용 방법**
1. 결정이 필요한 상황 인식
2. 대안들 나열
3. 각 대안의 장단점 분석
4. 결정 및 근거 기록
5. ADR 문서 생성 (ADR-XXX-title.md)

**측정 가능한 효과**
- 기술 부채 추적 가능
- 온보딩 시간 단축
- 재논의 방지

**적용 확대**
- [x] Level 2+ 프로젝트 필수
- [ ] ADR 템플릿 개선
